# design-tech-review

## 定位

用于作为 `dev-design-solution` 的高频增强，在现有方案基础上补做技术审视。

它只检查方案是否漏边界、漏异常、前后矛盾或描述不完整，不替代主设计流程，也不重新发明一套新方案。

默认不作为通用主入口；仅在 `dev-design-solution` 判断需要补洞，或用户明确点名该专项时使用。

## When to Use

适合以下场景：
- 已有候选方案或推荐方案，准备收敛前再做一轮补洞
- 方案主路径说得清楚，但担心异常路径、失败恢复或一致性遗漏
- 需要快速检查模块边界、状态流转、依赖假设是否自洽
- 需要在进入开发前确认方案已达到“可实现、可验证”的完整度

不适合以下场景：
- 需求边界仍不清，此时先回到 `dev-clarify-task`
- 方案方向尚未形成，此时先用 `dev-design-solution`
- 需要正式多轮评审产物或重型评审流程

## 检查骨架

围绕当前方案重点检查：
- 边界：模块职责、上下游责任、非目标是否清楚
- 异常路径：失败场景、超时、回退、降级是否覆盖
- 一致性：状态变化、数据写入、缓存或消息语义是否前后一致
- 完整性：关键章节、约束、验证思路、发布前提是否缺失

## 输出要求

输出应保持轻量，至少说明：
- 已检查哪些维度
- 发现了哪些缺口或矛盾
- 哪些问题必须先补齐
- 哪些问题只是提醒，不阻塞进入开发

## 使用原则

- 只在现有方案上补检查，不改写主设计结论
- 先指出缺口，再判断是否影响落地
- 重点关注高频漏项，不展开成长篇评审文档
- 若发现方案方向本身不成立，再回退 `dev-design-solution`

## 交接建议

- 检查通过后，回到 `dev-design-solution` 收敛最终方案
- 方案已可实施时，转入 `dev-build-change`
