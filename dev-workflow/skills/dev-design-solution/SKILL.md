# dev-design-solution

## 定位

用于在需求已基本明确后，结合代码上下文快速形成可落地的实现方案。

它是方案设计阶段的主入口：默认给出 2-3 个候选方案、收敛推荐方案，并输出模块影响、主要风险与验证思路；不内置重型评审回合、复杂分档或固定产物路径。

## 默认产物目录

如需把方案文档落盘，默认放到 `/home/joney/docs/requirements-development/<requirement-name>/solution-design/`，建议文件名使用 `solution.md`。

- `<requirement-name>` 使用简短、稳定、可读的需求名目录标识
- 用户明确指定路径时，优先使用用户路径
- 只需口头收敛方案时，可不创建文件

## When to Use

适合以下场景：
- 已有相对清晰的需求边界，需要决定怎么实现
- 需要比较 2-3 种实现路径，再收敛推荐方案
- 需要识别受影响模块、可复用模式、主要技术风险
- 编码前想先得到实现骨架和验证思路

不适合以下场景：
- 需求本身仍然模糊，边界和验收标准不清，此时先转 `dev-clarify-task`
- 只是做深度、正式、专项化的技术评审，此时可选增强调用 `design-tech-review`

## Workflow

### 1. 确认输入
至少确认以下内容：
- 需求目标与范围
- 验收标准或成功判断方式
- 可读取的代码上下文、现有模块或相似实现

如果输入不足，先指出缺口，不直接编造方案细节。

### 2. 分析代码上下文
聚焦最小必要分析：
- 相关模块、调用链和边界位置
- 现有可复用模式、约定和限制
- 可能受影响的数据结构、接口、状态流或配置

### 3. 生成候选方案
通常给出 2-3 个方案，并对比：
- 改动范围
- 实现复杂度
- 与现有模式的一致性
- 风险和后续演进成本

### 4. 收敛推荐方案
明确：
- 推荐哪一个方案
- 为什么它更适合当前需求和代码现状
- 为什么不选其他方案

### 5. 补齐落地信息
至少补齐以下内容：
- 受影响模块或文件范围
- 关键实现步骤或改动骨架
- 主要异常路径、边界条件和失败模式
- 验证思路：优先验证哪里、怎么判断方案成立

## 设计原则

- 先尊重现有代码模式，再考虑新增抽象
- 优先给出当前需求下的最小可行方案，避免为假设中的未来需求过度设计
- 方案要覆盖正常路径，也要显式考虑异常路径、一致性与完整性
- 所有结论尽量基于需求和代码证据，而不是空想扩展

## 输出物

默认输出一份轻量方案结论，建议包含：
- 背景与目标：本次设计要解决什么问题
- 候选方案对比：2-3 个方案的优缺点
- 推荐方案：结论与理由
- 模块影响：涉及的模块、接口、数据、状态或配置
- 风险与权衡：技术风险、兼容性风险、实施风险
- 验证思路：最小验证路径、重点回归点、失败信号
- 开放问题：仍需确认的关键点

## 结果要求

一份合格的结果应满足：
- 能指导后续编码，而不只是停留在概念层
- 能解释为什么选这个方案，而不是只罗列选项
- 能让开发者快速看出改动边界、风险和验证重点
- 保持轻量，不依赖固定目录、超长模板或多轮评审机制

## 可选增强

- 如需要做深度方案评审、系统性补漏洞或正式评审结论，可调用 `design-tech-review`

## 交接建议

- 方案已收敛且可以实施时，交给开发阶段主入口继续执行
- 若方案争议仍集中在需求边界而非实现本身，回退到 `dev-clarify-task`
