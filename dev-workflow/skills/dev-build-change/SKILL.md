# dev-build-change

## 定位

用于在代码开发阶段把“要改什么”落实成最小、可交付的代码变更。

它是开发阶段的主入口：先理解现有代码和边界，再实施最小安全改动；不内置重型流程、固定产物路径或特定技术栈细节。

## 默认产物目录

代码改动默认直接落在当前工作区源码中；如需额外落盘开发摘要，默认放到 `/home/joney/docs/requirements-development/<requirement-name>/code-implementation/`，建议文件名使用 `change-summary.md`。

- `<requirement-name>` 使用简短、稳定、可读的需求名目录标识
- 用户明确指定路径时，优先使用用户路径
- 若只有代码改动且无需额外摘要，可不创建该文件

## When to Use

适合以下场景：
- 需求和方案已基本明确，准备开始改代码
- 需要先理解现有实现、调用链和约定，再决定怎么改
- 目标是做最小安全改动，而不是顺手重构一大片
- 需要在改动后做最小必要自检，确认改动可继续推进

不适合以下场景：
- 需求边界仍模糊，此时先转 `dev-clarify-task`
- 实现路径还未收敛、需要先比较方案，此时先转 `dev-design-solution`
- 当前重点是验证结果而非继续改代码，此时转 `dev-verify-change`

## Workflow

### 1. 对齐前提
至少确认：
- 需求目标、完成标准已基本明确
- 本次明确不做的内容
- 可以读取的代码上下文、相关模块或相似实现

如果这些前提仍不成立，先回到 `dev-clarify-task` 或 `dev-design-solution`，不直接凭猜测改代码。

### 2. 先读懂再改
围绕本次任务做最小必要分析：
- 相关模块、入口、调用链和数据流
- 现有命名、分层、错误处理和复用模式
- 真正需要修改的位置，以及可能受影响的边界
- 仓库的隐性约定：测试发现方式、目录约束、日志粒度、配置开关使用习惯、注释密度
- 是否已经存在相似实现；若已有相似模式，默认沿用，不主动发明新结构

### 3. 做最小必要改动
编码时只保留当前任务必需的变更：
- 优先复用现有实现和模式，避免新增无必要抽象
- 保持接口、行为和风险边界清晰
- 不顺手做无关重构、清理或风格统一
- 新增或迁移单测时，若当前应用已有 `mock` 测试子目录或测试 suite 只扫描 `mock.*`，测试用例必须放到 `mock` 子目录下；只有仓库确实没有 `mock` 测试目录时，才按现有测试布局放置

复杂度控制按以下标准执行：

- **实现路径**
  - 默认优先单一路径实现；不要为同一语义同时维护串行/并行、老/新、A/B 两套主流程
  - 只有在以下情况才保留双路径：语义确实不同、顺序要求不同、故障隔离不同、灰度/回滚必须快速切换
  - 若新增双路径，必须能明确回答“为什么不能统一成一条路径”

- **方法拆分**
  - 以下情况适合拆方法：
    - 同一段逻辑在 `2` 个及以上位置复用
    - 一段逻辑本身是稳定、可命名的独立概念，而不是临时拼装步骤
    - 单个方法的有效代码行通常超过 `40-60` 行，且同时承担多个职责
    - 单个方法超过 `80` 行有效代码时，原则上必须评估拆分；若不拆，必须是单纯顺序编排、且拆开会损失可读性
    - 嵌套层级持续超过 `2-3` 层，拆出后能显著降低阅读负担
  - 以下情况不建议拆方法：
    - 只会被调用一次、且只是 `5-10` 行的薄包装
    - 拆出来后方法名只能重复代码表面动作，没有独立语义
    - 拆分只是为了“让方法变短”，却增加上下文跳转和参数透传

- **新增抽象**
  - 新增 helper、结果对象、上下文对象、额外类之前，先确认是否满足以下至少一项：
    - 会在 `2` 处及以上复用
    - 代表稳定业务概念，而不只是为了搬运几行代码
    - 能显著降低主流程复杂度，且不会引入更多透传参数
  - 如果新类型只是为了包 `2-4` 个临时变量、隐藏过多参数或挪走少量一次性逻辑，通常不值得新增

- **参数传递**
  - 能在使用点就地获取、就地计算、或从现有对象读取的信息，不要跨多层透传
  - 只有在以下情况才适合向下传参：
    - 值来自调用方决策，内层不应自行决定
    - 该值获取代价高，需要复用同一次计算/查询结果
    - 向下传递能避免重复 I/O、重复编译或重复构造
  - 单个方法若出现超过 `4` 个业务参数，或连续出现多个上下文透传参数，应重新评估接口设计
  - 不要为了“减少参数个数”而新增空心 context / DTO；先优先删参数、就地获取、或复用现有对象

- **判空与防御式代码**
  - 来自外部输入、配置、缓存、MQ、异步回调的数据，可以做防御式校验
  - 对内部强约束、不应为空的核心对象，优先暴露真实问题，不要层层补“理论上不会发生”的空判断
  - 额外判空、fallback、默认值分支，必须能说明真实触发场景；不能只因为“保险一点”

- **重复与冗余代码**
  - 遵循 `DRY` 与 `KISS`：相同或高度相似的逻辑在当前改动中若重复出现 `2` 次及以上，应优先合并，避免拷贝后小改
  - 已经在上游完成的判空、状态判断、权限判断、配置解析，不要在下游重复判断，除非中途语义发生变化
  - 多个分支若前置条件相同、后半段执行片段相同，优先合并条件或提取公共片段，而不是复制分支体
  - 同一方法内重复读取同一配置、重复查询同一对象、重复构造同一中间结果时，优先在局部复用一次结果
  - 不保留已经被统一逻辑覆盖的旧分支、旧变量、过渡代码、兼容占位写法；没有明确回滚诉求的临时代码应直接删除
  - 避免“拷贝一段代码再改两三行”的实现；若差异只在个别参数、条件或策略点，优先参数化、合并或就地整理
  - 以下重构手法可优先考虑：提前返回（guard clauses）、合并条件判断（consolidate conditional expression）、提取公共尾部/头部片段（consolidate duplicate conditional fragments）

- **开关、配置与日志**
  - 新增配置开关前，先确认是否真的存在环境差异、灰度需要或快速回滚诉求；否则优先固定实现
  - 若一个数值配置已天然覆盖布尔开关语义，不再重复新增布尔开关
  - `info` 日志只用于关键开始/结束、状态变化、低频且对排障有价值的事件
  - 高频循环、逐项处理、细粒度调试信息默认放 `debug`；只有数量天然很少且排障价值高时才保留 `info`

### 4. 补完善且准确的注释
最小改动不等于零注释；注释服务于人工查看代码，但注释密度默认跟随仓库现状。对非自解释逻辑、业务规则、兼容性约束和风险边界，应补充必要说明，让后续维护者能理解“为什么这样做”。

默认倾向：
- 默认少而准，优先补“没有注释就不容易读懂”的地方，而不是系统性铺满注释
- 注释重点解释业务意图、设计原因、约束来源和风险边界，而不是逐行翻译代码
- 注释必须与当前实现一致，不写猜测、过期背景或无法从上下文支撑的结论
- 修改已有复杂逻辑时，同步检查相邻注释是否仍准确，必要时一起更新
- 普通行内/块注释默认不用 `<p>`、`</p>` 等 HTML 标签；正式 JavaDoc 中若需要多段说明、IDE 提示空行会被忽略，或项目已有同类风格，可使用 `<p>` 分段

优先补充以下类型：
- 复杂业务规则、状态流转、特殊分支或历史兼容逻辑
- 非显而易见的边界条件、失败处理、幂等与并发保护
- 需要跨模块理解的调用约束、数据约定或外部依赖限制
- 新增公共方法、复杂参数或返回语义需要说明时的文档注释
- 影响后续排查、灰度、回滚、兼容或数据一致性的关键判断

避免以下噪音：
- 重复代码本身已经表达清楚的“翻译式注释”
- 为了注释而注释的泛泛说明
- 与当前实现容易失配的长篇设计解释
- 不能从代码、需求或上下文验证的猜测性说明

### 5. 编码后先做一次简化回看
进入验证前，至少自查一次：
- 是否保留了本可统一的一条主路径
- 是否有可下沉到使用点的透传参数或中间结果
- 是否新增了其实可以删除的 helper、结果对象、构造器、配置开关、空判断
- 是否存在重复判空、重复条件、重复分支体、重复配置读取、重复中间变量构造
- 是否有只为“显得工程化”而存在的拆分
- 是否有不符合仓库风格的注释或日志

### 6. 交给验证阶段
当改动已形成可验证结果后：
- 说明本次改了什么与影响范围
- 标出主要风险点
- 将最小验证重点交给 `dev-verify-change`

## 开发原则

- 先理解上下文，再修改代码
- 以最小改动解决当前问题，避免过度设计
- 尊重现有代码模式，优先复用而非重写
- 先控制影响范围，再考虑增强和扩展
- 复杂度控制和正确性同样重要；能少一层抽象、少一层透传、少一条路径时，默认优先更简单的方案
- 同样重视简洁与去重；若能通过合并分支、复用局部结果、删除重复判断来让实现更直接，默认优先更直接的写法
- 注释以“够用且准确”为标准，默认跟随仓库密度，不主动制造注释噪音
- 结论基于代码证据，不基于想象中的理想结构

## 输出物

默认输出一份轻量开发结果，至少包括：
- 改动摘要：本次完成了什么改动
- 受影响范围：涉及的模块、文件或接口边界
- 关键判断：为何这样改，复用了什么现有模式
- 注释说明：哪些非显而易见逻辑已补充注释，或为什么无需新增注释
- 风险提示：需要关注的边界、异常路径或兼容点
- 验证建议：下一步最小应验证什么

## 结果要求

一份合格的开发结果应满足：
- 能说明改动建立在已理解代码上下文之上
- 能证明改动范围受控且与任务目标一致
- 能说明新增抽象、方法拆分、参数传递和日志/注释选择是有依据的，而不是随手工程化
- 能自然衔接到验证阶段，而不是停在“已写完代码”
- 保持轻量，不携带特定栈的重型实现细节

## 可选增强

- 如任务复杂、需要拆分执行，或需要借助多后端子代理处理较大改动，可调用 `build-codeagent`

## 交接建议

- 改动完成后，进入 `dev-verify-change` 做围绕本次变更的最小有效验证
- 如果在编码前发现实现路径仍未收敛，回退到 `dev-design-solution`
