# verify-java-coverage

## 定位

用于作为 `dev-verify-change` 的 Java 覆盖率深度增强，处理 Maven、JaCoCo 与跨模块覆盖率场景。

它聚焦覆盖率证据与补测判断，不替代主验证流程里的最小验证结论。

默认不作为通用主入口；仅在 `dev-verify-change` 判断需要覆盖率深挖，或用户明确点名 Java 覆盖率时使用。

## When to Use

适合以下场景：
- Java / Maven 项目已经完成改动，需要进一步判断覆盖率是否足够
- 变更跨多个模块，基础验证无法准确反映真实覆盖情况
- 需要做覆盖率门控，而不是只看测试是否通过
- 需要定位未覆盖行、未覆盖方法，并给出补测建议

不适合以下场景：
- 当前还没完成最小测试验证，此时先用 `dev-verify-change`
- 项目不是 Java / Maven / JaCoCo 体系
- 只需快速判断功能是否通过，不需要深挖覆盖率

## 检查骨架

重点检查：
- 模块结构：测试代码与生产代码是否跨模块分离
- 覆盖率来源：JaCoCo 配置是否可用，报告是否可信
- 临时方案：仓库未配置 JaCoCo 时，优先用命令行 `-javaagent` / JaCoCo CLI 生成报告；确需临时改 `pom` 时，跑完必须删除或还原
- 变更粒度：优先看本次改动涉及的类、方法和变更行
- 门控结论：新增/修改生产代码的 JaCoCo 可执行行覆盖率必须大于 80%，未达标要列出缺口
- 补测方向：优先补哪些分支、异常路径和边界条件

## 使用原则

- 先有基础验证结论，再补覆盖率深挖
- 优先围绕本次变更收集覆盖率证据，不扩大成全仓普查
- 跨模块时优先确认报告是否遗漏真实生产类
- 不为验证覆盖率永久修改仓库 `pom`；临时配置必须在最终交付前清理干净
- 不用全仓覆盖率或整类覆盖率替代改动行覆盖率；整类覆盖率只能作为辅助风险说明
- 门控失败时给出补测建议，而不只报一个百分比
- 若覆盖率数据本身不可信，先说明证据限制，再限制结论范围

## 输出要求

输出至少包括：
- 覆盖率检查范围与模块结构判断
- 覆盖率生成方式，尤其是是否使用了临时 JaCoCo 配置以及是否已清理
- 报告是否可信，是否存在跨模块遗漏风险
- 关键覆盖率结论与门控结果
- 未覆盖重点与补测建议

## 交接建议

- 覆盖率结论回交 `dev-verify-change` 汇总整体验证结论
- 若发现问题本质是实现遗漏，回到 `dev-build-change`
