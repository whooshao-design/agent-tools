# lang-java-service-patterns

## 定位

用于作为 `dev-build-change` 和 `dev-review-change` 的 Java 服务开发高频模式增强。

它沉淀 Java 服务代码里最常复用的实现与审查关注点，帮助开发和评审快速对齐，而不是展开成一份完整语言手册。

默认不作为通用主入口；仅在主开发或评审流程判断需要 Java 服务实现模式校准，或用户明确点名该专题时使用。

## When to Use

适合以下场景：
- 正在开发或评审 Java 服务层、应用层、领域服务等代码
- 需要快速检查幂等、异常处理、事务边界是否符合常见服务模式
- 需要判断某段服务代码是否职责混乱、过度防御或测试模式失衡
- 需要补齐服务层常见测试思路，而不是从零设计测试策略

不适合以下场景：
- 当前不是 Java 服务代码场景
- 需要数据库、中间件专项深挖，此时转对应 review 增强 skill
- 需要完整语言规范或框架百科式资料

## 关注模式

重点关注：
- 幂等：写操作、重试入口、重复请求处理是否自洽
- 异常：边界处是否翻译异常，内部是否避免吞异常或失真包装
- 事务：事务范围是否收敛，是否混入不适合放进事务的远程调用
- 分层：Service、DAO、Adapter、Controller 职责是否清晰
- 测试：是否覆盖正常路径、异常路径、边界条件和关键分支

## 使用原则

- 优先遵循项目现有 Java 服务模式，而不是引入新流派
- 先判断是否影响正确性和可维护性，再讨论风格偏好
- 对事务、幂等和异常问题优先提级，因为它们最容易造成服务事故
- 测试建议围绕当前改动给出，不扩展成全量重写
- 只保留高频模式，不复制框架官方文档

## 输出要求

输出至少包括：
- 当前代码涉及的 Java 服务模式判断
- 发现的幂等、异常、事务或分层问题
- 对应的测试补充方向
- 哪些问题影响实现，哪些问题影响评审结论

## 交接建议

- 开发阶段使用时，结果回交 `dev-build-change`
- 评审阶段使用时，结果回交 `dev-review-change`
