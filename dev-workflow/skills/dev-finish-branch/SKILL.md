# dev-finish-branch

## 定位

用于在提交、提测、交付或请他人接手前，对当前开发分支做最终收口。

它是交付前阶段的主入口：确认 diff 是否清晰、验证是否到位、评审问题是否已处理，并整理可直接交付的摘要；不替代验证阶段重跑完整测试，也不重复做完整代码评审。

## 默认产物目录

如需把交付摘要落盘，默认放到 `/home/joney/docs/requirements-development/<requirement-name>/delivery-handoff/`，建议文件名使用 `handoff.md`。

- `<requirement-name>` 使用简短、稳定、可读的需求名目录标识
- 用户明确指定路径时，优先使用用户路径
- 只需在对话中给出收口结论时，可不创建文件

## When to Use

适合以下场景：
- 本次改动已基本完成，准备提交、提测、交付或发起后续流程
- 需要确认当前 diff 是否只包含应交付内容
- 需要检查验证结论、评审结论与已知风险是否已经收口
- 需要输出一份简明交付摘要，方便自己或他人继续推进

不适合以下场景：
- 代码仍在频繁修改，范围尚未稳定，此时先回到 `dev-build-change`
- 关键验证尚未完成，无法判断是否可交付，此时先转 `dev-verify-change`
- 尚未做过基本代码评审，但已需要评审结论支撑交付，此时先转 `dev-review-change`

## Workflow

### 1. 收口当前 diff
先看本次准备交付的改动是否清晰：
- 改了什么
- 为什么改
- 是否混入无关变更
- 是否还留有明显未完成内容

### 2. 确认验证状态
只确认现有验证状态是否足以支撑交付：
- 已执行了哪些验证
- 关键路径是否已有结果
- 当前失败、跳过或未覆盖点是什么
- 是否仍存在必须补验的风险

这里关注“是否已满足当前交付条件”，而不是重新展开完整验证流程。

### 3. 确认评审状态
检查评审相关结论是否已收敛：
- 是否已有代码评审结论
- 阻断问题是否已修复或明确处理
- 可延后问题是否已被记录或接受
- 当前是否仍存在不能忽略的发布风险

### 4. 整理交付摘要
把当前分支收敛成可交付信息：
- 变更摘要
- 验证摘要
- 风险摘要
- 后续注意事项

### 5. 给出最终收口判断
最终明确：
- 是否适合提交或交付
- 若暂不适合，卡点在哪里
- 若可以推进，交付时需要同步哪些信息

## 收口原则

- 以当前分支现状为准，不把收口阶段扩成新一轮开发
- 优先确认差异、验证、评审三件事是否闭环
- 不重跑完整测试，不重复做完整 CR
- 对未解决风险要显式标出，不把未知写成已完成
- 输出要让接收方一眼看懂本次交付的边界与状态

## 输出物

默认输出一份轻量收口结果，建议包含：
- 交付范围：本次准备提交或交付的内容
- diff 结论：是否存在无关改动、遗留项或需清理内容
- 验证状态：已验证、未验证、失败或带风险接受的部分
- 评审状态：已处理问题、待跟进问题、已接受风险
- 交付摘要：可直接用于提交说明、提测说明或 handoff

## 结果要求

一份合格的收口结果应满足：
- 能说明当前分支是否真的到了可交付状态
- 能区分“已完成”“已知风险”“仍未收口”
- 能帮助后续提交、提测或交接，而不是重复前面阶段的正文
- 保持轻量，不重新展开完整测试或评审流程

## 可选增强

- 若发现关键验证结论不足，回到 `dev-verify-change`
- 若发现仍有重要评审问题未收敛，回到 `dev-review-change`

## 交接建议

- 可提交时，直接基于本结果整理 commit、提测或交付说明
- 不可提交时，按卡点回退到对应阶段继续处理
