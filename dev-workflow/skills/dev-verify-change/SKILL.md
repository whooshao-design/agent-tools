# dev-verify-change

## 定位

用于在代码改动后，以最小有效成本确认这次变更是否成立。

它是验证阶段的主入口：围绕本次改动优先做最小必要验证，先看结果是否通过、失败点在哪里、是否需要补充覆盖率或回归；不内置特定技术栈的重型实现细节。

## 默认产物目录

如需把测试验证结果落盘，默认放到 `/home/joney/docs/requirements-development/<requirement-name>/test-verification/`，建议文件名使用 `verification.md`。

- `<requirement-name>` 使用简短、稳定、可读的需求名目录标识
- 用户明确指定路径时，优先使用用户路径
- 只需在对话中汇报验证结论时，可不创建文件

## When to Use

适合以下场景：
- 代码已完成改动，需要判断本次变更是否可接受
- 需要执行与本次改动最相关的测试、构建或检查
- 需要先确认失败是否由本次变更引起，再决定是否扩大验证范围
- 需要给出明确验证结论，而不是只罗列命令输出

不适合以下场景：
- 当前还在理解代码或继续修改代码，此时先回到 `dev-build-change`
- 需求或方案尚未稳定，验证范围无法界定
- 需要做专项深挖但尚未完成基础验证结论

## Workflow

### 1. 明确验证边界
至少确认：
- 本次改动了什么
- 哪些模块、接口、页面或行为受到影响
- 最小可证明“改动成立”的验证方式是什么

### 2. 先做最小有效验证
优先执行与本次改动最相关的检查，例如：
- 直接受影响模块的测试
- 最小必要的构建或编译检查
- 最贴近变更路径的手工或自动验证

原则是先小后大，先验证核心路径，再决定是否扩展。

在真正执行命令前，先识别仓库级测试约束：
- 对 Maven / TestNG 仓库，先检查 `pom.xml`、Surefire 配置和仓库文档，确认是否强依赖 `suiteXmlFile`、固定 suite XML、profile 或额外 JVM 参数
- 若仓库已要求 `suiteXmlFile`，优先复用仓库已有 suite，或显式传入 `-DsuiteXmlFile=...`，不要为了让临时命令跑通就先建议修改 `pom`
- 新增、修改或迁移单测时，必须先检查测试源码目录下是否已有 `mock` 目录；只要存在，所有新增或修改的单元测试用例都必须放到 `mock` 目录的子目录或对应 `mock.*` 包下，否则验证不能算通过
- 只有仓库确实没有 `mock` 测试目录时，才接受沿用原测试布局；如果测试 suite 只扫描 `mock.*`，还必须确认测试类会被 suite 实际执行
- 若用户级说明、仓库 `AGENTS.md` 或模块文档已经给了推荐测试命令，优先沿用并在验证结果里写清楚具体命令，避免重复踩同一类运行方式问题

代码变更必须包含覆盖率检查：
- Java / Maven 项目优先调用 `verify-java-coverage`，用 JaCoCo 或仓库已有覆盖率工具生成本次相关测试的覆盖率证据
- 覆盖率门禁默认看新增/修改生产代码的 JaCoCo 可执行行覆盖率，必须大于 90%；不要用全仓或整类覆盖率替代改动行覆盖率结论
- 如果仓库没有 JaCoCo 配置，允许临时增加最小 JaCoCo 配置或通过命令行临时加载插件；覆盖率跑完后必须删除或还原临时配置，不能把临时验证配置混入最终代码改动
- 如果环境、依赖或权限导致覆盖率无法生成，验证结论不能写成完全通过；必须明确阻塞原因、已执行的替代验证和下一步可执行命令

### 3. 解释结果并收敛结论
拿到结果后明确区分：
- 通过：哪些关键路径已被验证
- 失败：失败发生在哪里、是否明显与本次改动相关
- 未覆盖：哪些风险点尚未被验证到

### 4. 按需增强
覆盖率属于代码变更的必跑验证；相邻回归或专项验收仍按需追加。只有在基础测试和覆盖率都足以支撑结论时，才直接给出是否可进入评审、提测或交付的结论。

## 验证原则

- 围绕本次改动做最小有效验证，不默认扩大成全量检查
- 先遵循仓库级测试入口约束，再决定怎么执行验证命令
- 若测试源码目录已有 `mock` 目录，所有新增或修改的单元测试用例必须位于 `mock` 目录的子目录或对应 `mock.*` 包下
- 代码变更必须执行覆盖率检查，并说明报告位置、覆盖范围和未覆盖风险
- 覆盖率必须优先给出新增/修改生产代码可执行行的覆盖率与是否超过 90%
- 先定位失败，再决定修复还是扩测
- 验证结论必须能支撑下一步决策
- 回归检查是按需增强，不是默认重流程
- 保持与项目现有测试和验证方式一致

## 输出物

默认输出一份轻量验证结果，至少包括：
- 验证范围：本次验证覆盖了哪些改动点
- 执行结果：哪些检查通过、哪些失败、哪些未执行
- 单测布局：是否存在 `mock` 测试目录；新增或修改的单测是否全部放在 `mock` 子目录或对应包下
- 覆盖率结果：报告位置、覆盖范围、关键未覆盖点和是否达标
- 失败分析：失败是否与本次改动直接相关
- 验证结论：通过、需修复后重验，或带风险接受
- 后续建议：是否需要补充覆盖率、回归或专项验收

## 结果要求

一份合格的验证结果应满足：
- 能说明验证是围绕本次改动展开，而不是泛泛跑一遍
- 能明确测试是否通过，以及通过结论建立在哪些证据上
- 若存在 `mock` 测试目录，能明确新增或修改单测的目录/包位置符合要求
- 能明确覆盖率是否已检查；若没有覆盖率报告，不能把未知当作通过
- 新增或修改代码行覆盖率未超过 90% 时，不能把验证结论写成达标
- 保持轻量，不内嵌 Maven、JaCoCo、浏览器自动化等重实现正文

## 可选增强

- Java / Maven / JaCoCo 覆盖率验证必须调用 `verify-java-coverage`
- 如需要做浏览器、页面或交互验收，可调用 `verify-browser-qa`
- 如验证显示问题仍在代码实现本身，回到 `dev-build-change`

## 交接建议

- 验证通过且风险可接受时，交给后续评审或收口阶段
- 如验证显示问题仍在代码实现本身，回到 `dev-build-change` 继续修正
