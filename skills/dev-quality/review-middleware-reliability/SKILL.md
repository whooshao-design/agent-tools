# review-middleware-reliability

## 定位

用于作为 `dev-review-change` 的中间件可靠性专项增强，补查缓存、消息、调度和外部依赖调用的稳定性问题。

它关注可靠性与可恢复性，不替代主评审流程里的通用正确性判断。

默认不作为通用主入口；仅在 `dev-review-change` 判断需要可靠性专项深挖，或用户明确点名中间件风险评审时使用。

## When to Use

适合以下场景：
- 变更涉及 Redis、MQ、延迟任务、定时任务或外部服务调用
- 需要确认超时、重试、幂等、降级或补偿策略是否合理
- 需要评估消费失败、重复投递、缓存失效或依赖抖动时的行为
- 需要判断是否缺少死信、熔断、限流或关键埋点

不适合以下场景：
- 改动完全不涉及中间件或外部依赖可靠性
- 还没形成基础 diff 和验证证据
- 只是一般代码风格问题，不属于可靠性专题

## 检查骨架

重点检查：
- Redis：key 设计、TTL、失效策略、一致性与热点风险
- MQ：生产投递、消费确认、重复消息、死信与重试语义
- 超时与重试：是否有边界，是否会放大故障
- 幂等与降级：失败重放是否安全，依赖异常时如何退化
- 可观测性：关键入口、失败路径、重试和丢弃是否可见

## 使用原则

- 优先判断故障时系统会怎样表现，而不只看正常路径
- 重试必须与幂等一起看，不能孤立判断
- 缓存和消息问题优先看一致性与恢复性
- 对会放大发布风险或运行时事故的点提高严重度
- 若只是一般实现细节，交回主评审处理，不在此展开

## 输出要求

输出至少包括：
- 涉及的中间件或外部依赖范围
- 超时、重试、幂等、降级等关键判断
- 故障场景下的主要风险与恢复建议
- 哪些问题会影响上线稳定性

## 交接建议

- 专项结论回交 `dev-review-change`
- 若专项问题需要改实现，回到 `dev-build-change`
