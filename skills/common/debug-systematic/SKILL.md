# debug-systematic

## 定位

跨语言、跨项目的系统化调试方法论。它治理"怎么查问题"的过程，不绑定具体工具；
查 Java 服务器用 `java-server-diagnostics`，查数据用 `env-access` 类 skill，本 skill 负责让这些动作有章法。

## 核心原则

- **先复现，后修复**：不能稳定复现的问题不要急着改代码；改了也无法证明修好了。
- **一次只验证一个假设**：每个动作前先写下"我认为原因是 X，如果是 X 那么观察 Y 应该看到 Z"；观察结果与预期不符就放弃该假设，不带着旧假设继续改。
- **二分缩小范围**：在调用链、时间线、数据集、commit 历史（`git bisect`）上做二分，而不是逐行通读。
- **保留证据链**：每个阶段记录关键证据（日志片段、traceId、复现步骤、对比结果），结论必须能被证据支撑。
- **不顺手重构**：调试过程中发现的"丑代码"记下来，修复只做最小改动；重构是另一个任务。

## 四阶段流程

### 1. 复现（Reproduce）

- 明确"预期行为 vs 实际行为"，一句话写清差异。
- 找最小复现路径：能用单测复现就不用接口，能用接口复现就不用页面。
- 偶现问题先收集发生条件：时间窗口、特定数据、并发量、特定机器；无法复现时转为加观测点等待下次发生。

### 2. 定位（Isolate）

- 沿调用链二分：在中点加观测（日志/断点/临时断言），判断问题在前半段还是后半段，重复直到收敛到单个函数或语句。
- 时间维度二分：如果"以前是好的"，用 `git bisect` 或按发布版本回溯，找引入问题的变更。
- 数据维度二分：用最小数据集复现，逐步删减输入直到找到触发问题的最小数据。

### 3. 根因（Root Cause）

- 区分"出错位置"和"根因位置"：异常抛出点往往只是受害者，向上追溯到第一个状态被破坏的地方。
- 连续问 3-5 个"为什么"，直到答案是设计决策、边界遗漏或环境差异，而不是另一个现象。
- 根因必须能解释**全部**已观察到的现象；解释不了的现象说明根因不完整或有多个问题叠加。

### 4. 验证（Verify）

- 修复后用第 1 阶段的复现路径确认问题消失。
- 为根因补一个会失败的测试（修复前红、修复后绿），防止回归。
- 检查同类位置是否有相同隐患（同模式的代码、同来源的数据）。
- 清理调试期间加的临时日志、断言和开关。

## 停止与升级

- 同一假设验证失败 3 次，或 2 小时无进展：停下来，把已收集的证据和已排除的假设整理成清单，换角度重新提假设或求助。
- 发现问题涉及环境/基础设施（机器、网络、中间件配置），转交对应排障 skill 并附上证据链。

## 输出要求

调试结束时给出：现象、复现路径、根因（含证据）、修复方式、验证结果、是否有同类隐患。
不要只说"已修复"。
