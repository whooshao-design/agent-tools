# dev-auto-loop

## 定位

用于在开发任务已具备自动推进条件时，自动串联多个开发阶段并持续迭代，直到达到停止条件。

它是自动编排阶段的主入口：负责组织 `dev-build-change`、`dev-verify-change`、`dev-review-change`、`dev-finish-branch` 的执行顺序与循环策略；不复制各阶段 skill 的详细正文，也不承载技术栈专项细节。

## 默认产物目录

如需把循环状态落盘，默认放到 `/home/joney/docs/requirements-development/<requirement-name>/auto-loop/`，建议文件名使用 `status.md`。

- `<requirement-name>` 使用简短、稳定、可读的需求名目录标识
- 用户明确指定路径时，优先使用用户路径
- 只需在对话中逐轮汇报状态时，可不创建文件

## When to Use

适合以下场景：
- 任务目标和边界已基本清晰，适合自动推进多轮开发闭环
- 需要把“修改 → 验证 → 复查 → 收口”串成连续流程，而不是手动一步步切换
- 希望在满足明确停止条件前，自动继续下一轮修复或验证
- 任务规模较大，必要时需要调用 `build-codeagent` 协助实现或拆分执行

不适合以下场景：
- 需求边界仍然模糊，自动循环会放大误判，此时先转 `dev-clarify-task`
- 方案仍未收敛，尚不适合自动持续编码，此时先转 `dev-design-solution`
- 只需要执行单个阶段，不需要跨阶段编排

## Workflow

### 1. 判断是否适合自动编排
先确认以下前提是否成立：
- 任务目标、范围和完成标准基本清晰
- 当前已有可执行的代码上下文和操作路径
- 可以定义本轮循环的停止条件与上限

如果这些前提不成立，先退回对应主阶段，不直接启动自动循环。

### 2. 建立循环骨架
自动编排默认围绕以下骨架组织：
- 先进入 `dev-build-change`
- 产出可验证结果后进入 `dev-verify-change`
- 验证通过或结论足够后进入 `dev-review-change`
- 达到交付条件时进入 `dev-finish-branch`

必要时可根据结果回退到前一阶段继续下一轮，而不是线性只走一遍。

### 3. 定义停止条件与上限
开始前明确：
- 什么情况下视为本轮任务完成
- 什么情况下应停止并交还人工判断
- 最大循环轮次是多少
- 哪些失败会触发回退或终止

### 4. 每轮输出状态摘要
每完成一轮，都要收敛成简明状态：
- 本轮做了什么
- 当前结论是什么
- 剩余卡点是什么
- 下一轮准备处理什么

### 5. 失败时回退或升级
当自动推进受阻时，按情况处理：
- 实现问题明显 → 回退 `dev-build-change`
- 验证证据不足或失败待定位 → 回退 `dev-verify-change`
- 风险判断未收敛 → 回退 `dev-review-change`
- 若任务过大、上下文过重或需要并行实现，可调用 `build-codeagent`

## 编排原则

- 自动编排只负责串阶段，不替代各阶段自身判断
- 每轮都要有明确状态结论，避免静默长循环
- 必须有循环上限，避免无限推进
- 必须有停止条件，无法继续时及时回退或转人工判断
- 仅在需要时调用增强能力，不把所有专项能力塞进主入口

## 输出物

默认输出一份轻量循环摘要，建议包含：
- 当前轮次：已完成到第几轮
- 阶段轨迹：本轮经过了哪些主阶段
- 本轮结果：通过、回退、继续或终止
- 当前卡点：尚未解决的问题或风险
- 下一步动作：继续哪一阶段，或交给人工决策
- 最终结论：已完成 / 达到上限停止 / 因失败终止

## 结果要求

一份合格的自动编排结果应满足：
- 能看出当前是如何在多个主阶段之间推进的
- 能说明为什么继续、为什么回退、为什么停止
- 每轮都有摘要，最终能沉淀出清晰状态
- 保持轻量，不复制开发、验证、评审、收口阶段的详细规则

## 可选增强

- 如需要较大规模实现、并行处理或额外子代理协作，可调用 `build-codeagent`

## 交接建议

- 若循环已达到可交付状态，进入 `dev-finish-branch`
- 若循环在某一阶段持续受阻，回退到对应主入口由人工继续判断
