# review-db-change

## 定位

用于作为 `dev-review-change` 的数据库专项增强，补做数据库相关风险审查。

它聚焦 SQL、DDL、索引、兼容性和发布顺序，不替代通用代码评审主流程。

默认不作为通用主入口；仅在 `dev-review-change` 判断需要数据库专项深挖，或用户明确点名数据库评审时使用。

## When to Use

适合以下场景：
- 评审变更中包含 SQL、Mapper、Repository 或数据库访问路径调整
- 涉及表结构、索引、字段语义或事务边界变化
- 需要判断数据库变更是否兼容现网数据和发布顺序
- 需要识别查询性能、锁竞争或数据迁移风险

不适合以下场景：
- 变更与数据库无关
- 还未完成基础代码评审证据收集
- 只是讨论抽象数据模型，没有真实数据库落点

## 检查骨架

重点检查：
- SQL 正确性：条件、分页、排序、更新范围是否正确
- DDL 兼容性：新增、删除、改类型、默认值、可空性是否安全
- 索引影响：是否命中高频条件，是否引入锁表或慢查询风险
- 事务边界：是否过大、是否跨不该放进事务的外部操作
- 发布顺序：DDL、代码、回滚和数据迁移步骤是否自洽

## 使用原则

- 先基于本次变更证据判断，不做数据库全量体检
- 优先识别会影响正确性和发布安全的点
- 兼容性问题优先于“写法是否优雅”
- 对高风险结论要明确影响范围和建议发布顺序
- 若问题只属于一般实现模式，再交回主评审结论处理

## 输出要求

输出至少包括：
- 涉及的表、SQL 或 DDL 范围
- 发现的兼容性、事务或索引风险
- 建议的发布顺序或回滚注意点
- 哪些问题必须先修，哪些可带风险跟进

## 交接建议

- 专项结论回交 `dev-review-change` 汇总最终评审结论
- 若发现问题源于实现方式，回到 `dev-build-change`
