# dev-clarify-task

## 定位

用于把“想做什么”澄清成“这次具体做什么、不做什么、做到什么算完成”。

它是需求澄清阶段的主入口，只保留高频骨架，不负责技术方案评审、固定文档落盘或重型流程编排。

## 默认产物目录

如需把需求澄清结果落盘，默认放到 `/home/joney/docs/requirements-development/<requirement-name>/requirements/`，建议文件名使用 `summary.md`。

- `<requirement-name>` 使用简短、稳定、可读的需求名目录标识
- 用户明确指定路径时，优先使用用户路径
- 只需在对话中给出结果时，可不创建文件

## When to Use

适合以下场景：
- 用户描述偏口语、目标模糊、边界不清
- 需求里缺少验收标准、非目标或关键依赖
- 准备进入方案设计或编码前，需要先把范围收紧
- 发现“要改什么”已经大致明确，但“做到什么算完成”仍不明确

不适合以下场景：
- 需求边界、验收标准、依赖和非目标都已明确
- 用户已经要求直接做技术方案评估或多方案对比，此时转 `dev-design-solution`

## Workflow

### 1. 复述任务
- 先用一句话复述当前理解
- 明确目标对象、触发场景和核心结果
- 若复述都无法成立，先缩小问题再继续

### 2. 收敛边界
围绕最小可执行范围确认：
- 本次要交付的能力
- 本次明确不做的内容
- 必须满足的前置条件或外部依赖
- 需要保留的约束：时效、权限、兼容性、数据来源、失败处理

### 3. 明确验收标准
优先得到可判断“是否完成”的结果：
- 成功路径是什么
- 关键边界条件是什么
- 失败或异常时应有什么表现
- 哪些结果必须可验证

### 4. 暴露未知项
把无法立即确认的点单独列出：
- 缺失事实
- 待业务确认项
- 影响实现选择的关键假设

### 5. 输出可执行摘要
给出一份轻量结论，供后续设计或开发直接使用。

## 沟通原则

- 一次优先澄清一个最关键的不确定点
- 先确认范围，再谈实现
- 主动追问非目标，避免默认把后续迭代纳入本次范围
- 对模糊表述做显式改写，例如“更好用”要改写成可验收描述
- 无法确认时明确标记，不伪造确定性

## 输出物

默认输出一份轻量需求澄清结果，建议包含：
- 任务目标：一句话说明这次要解决什么问题
- In Scope：本次包含的能力边界
- Out of Scope：本次不包含的内容
- 验收标准：3-7 条可验证结果
- 关键依赖与约束：上下游、权限、数据、时序、兼容性等
- 开放问题：仍需确认但不应被忽略的点

## 结果要求

一份合格的结果应满足：
- 能支持进入 `dev-design-solution` 或直接编码
- 能明确区分“这次做”和“这次不做”
- 能用验收标准判断是否完成
- 不依赖超长模板或固定目录结构才能使用

## 交接建议

- 若需要基于代码上下文做 2-3 个实现方案对比，转 `dev-design-solution`
- 若需求已经足够清晰且用户要求直接改代码，可直接进入开发阶段主入口
