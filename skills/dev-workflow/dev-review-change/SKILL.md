# dev-review-change

## 定位

用于在代码改动与基础验证之后，对本次变更做证据驱动的代码评审。

它是代码评审阶段的主入口：围绕当前变更识别正确性、架构、安全与稳定性风险，输出问题判断与后续动作；数据库、中间件、语言实现等重专项细节下沉到增强 skill，不在这里展开。

## 默认产物目录

如需把代码评审结论落盘，默认放到 `/home/joney/docs/requirements-development/<requirement-name>/code-review/`，建议文件名使用 `review.md`。

- `<requirement-name>` 使用简短、稳定、可读的需求名目录标识
- 用户明确指定路径时，优先使用用户路径
- 只需在对话中给出评审结论时，可不创建文件

## When to Use

适合以下场景：
- 代码已改完，且已经具备最小可评审证据，如 diff、验证结果、关键上下文
- 需要判断这次改动是否可以继续合并、交付或进入收口
- 需要按正确性、架构、安全、稳定性几个维度检查本次变更
- 需要把问题按严重程度收敛，并给出可执行修复建议

不适合以下场景：
- 还在实现或调试过程中，代码尚未稳定，此时先回到 `dev-build-change`
- 连最小验证都还没做，缺少基础结果支撑判断，此时先转 `dev-verify-change`
- 需要数据库、中间件或特定语言栈的专项深挖，此时按需调用对应增强 skill

## Workflow

### 1. 明确评审对象
先确认本轮评审聚焦的是哪一批改动：
- 本次改动的目标与边界
- 变更涉及的模块、接口、数据或配置
- 已有验证结论与已知风险

### 2. 先拿证据再下结论
优先基于可见证据评审，例如：
- diff 与受影响文件上下文
- 已执行的测试、构建或验收结果
- 需求、方案或实现说明中的关键约束

若关键证据缺失，先明确缺口，再限制结论范围，而不是把猜测写成结论。

### 3. 按核心维度评审
默认围绕以下四类问题展开：
- 正确性：逻辑、边界、异常路径、状态一致性
- 架构：分层、职责、复用、改动边界是否失控
- 安全：输入处理、权限、敏感信息、危险用法
- 稳定性：失败处理、回退路径、可观测性、发布风险

默认还要检查代码设计是否“过重或冗余”，重点看：
- 是否为同一语义维护了多条主路径，而其实可以统一
- 方法拆分是否有明确收益，还是只增加跳转和参数透传
- 是否新增了缺乏复用价值的 helper、结果对象、context、构造器或开关
- 是否出现重复判空、重复条件、重复分支体、重复配置读取、重复对象构造
- 是否存在拷贝后小改的重复代码，而不是复用或合并
- 日志、注释、防御式代码是否超过仓库需要，造成噪音

评审这些点时，可优先参考通用经验：
- `DRY`：不要保留重复实现
- `KISS`：优先更直接、更少层级的实现
- Guard Clauses：用提前返回减少嵌套
- Consolidate Conditional Expression / Fragments：合并重复条件和重复分支片段

### 4. 收敛问题与建议
对发现的问题给出：
- 证据位置
- 问题判断
- 影响范围
- 严重程度
- 建议动作

严重程度默认按以下标准判定：

- **高**
  - 会直接导致功能错误、错误结果、状态不一致、数据错误、数据丢失、权限/安全风险
  - 会导致发布后高概率故障，或当前变更缺少安全上线所必需的保护
  - 会让现有验证结论失真，例如测试没有覆盖到真实路径、关键分支根本跑不到、核心回退路径不可用
  - 架构上虽然不一定立刻出错，但已把本次改动推到明显不可接受的复杂度，例如多条主路径并存且语义重复、关键状态靠隐式约定传递、容易产生重复执行/重复编译/重复副作用
  - 处理建议：原则上合并前必须修复

- **中**
  - 当前大概率还能工作，但存在较明显的正确性边界风险、维护风险或后续演化风险
  - 包括但不限于：低价值抽象、过多参数透传、重复判空、重复分支、重复代码、日志/注释/防御式代码明显偏重、方法拆分收益不足
  - 这类问题短期可能不阻断发布，但继续累积会显著抬高维护成本，或让后续改动更容易出错
  - 处理建议：优先在当前轮修复；若暂不修，需明确接受原因

- **低**
  - 不影响当前正确性、稳定性和上线安全，主要是局部可读性、一致性或轻微清理建议
  - 例如命名还能更贴切、日志文案可更统一、注释表述可更准确、局部实现还有进一步精简空间
  - 处理建议：可本轮顺手处理，也可后续跟进，不应阻断合并

判级时额外遵循：
- 不要把“能工作的代码风格建议”直接判成高
- 对重复代码、重复判空、过度拆分、低价值抽象，默认优先判中，而不是高；只有它们已经明显影响正确性或发布安全时才升高
- 高严重度必须明确说明“为什么会影响正确交付或安全上线”
- 若证据不足以证明问题一定成立，降低结论强度，写成风险而不是定性 bug

### 5. 输出评审结论
最终明确：
- 是否建议继续推进
- 哪些问题必须先处理
- 哪些问题可后续跟进
- 是否需要转专项增强 skill 深挖

推荐按以下模版输出，优先保证“先结论、再阻断、再证据、再动作”：

- **结论概览**
  - 一句话给出整体判断：`可继续推进` / `修复后再评` / `带风险接受`
  - 用 `1-3` 行说明结论依据，例如：是否存在高严重度问题、验证是否充分、是否有未关闭的上线风险

- **高严重度**
  - 只列真正阻断合并/上线的问题；没有则明确写“未发现”
  - 每个问题尽量保持固定结构：
    - `问题`：一句话说明是什么
    - `证据`：文件、位置、验证信号或行为证据
    - `影响`：为什么会影响正确交付或安全上线
    - `建议`：应如何修复或规避

- **中严重度**
  - 列出应优先处理的重要问题，例如低价值抽象、重复代码、过多透传、设计明显偏重
  - 每个问题同样给出 `问题 / 证据 / 影响 / 建议`

- **低严重度 / 建议**
  - 放不阻断交付的可读性、一致性、轻微清理项
  - 控制数量，避免把零散想法堆成噪音；没有则可省略

- **验证与证据**
  - 简要写清已参考的 diff、测试、覆盖率、构建结果和上下文范围
  - 若证据不足，要单列说明“哪些结论是受限判断”

- **后续动作**
  - 明确下一步：先修哪些问题、修完后补什么验证、是否需要专项复查

模版使用时遵循以下行业通用经验：
- Lead with recommendation：先给合并建议，不让读者先翻完整篇才能知道结论
- Blockers first：阻断问题优先，避免把一般建议淹没真正风险
- One issue, one evidence, one action：一条问题配一条证据和一条动作，避免泛泛而谈
- Separate fact from preference：事实性风险与风格偏好分开写，避免把个人偏好伪装成 bug
- Keep findings finite：默认优先列最重要的 `3-5` 个问题；轻微建议不要无限展开

## 评审原则

- 只评审本次变更，不把范围扩成对整个系统做全面审计
- 结论基于证据，不基于想象中的理想实现
- 先关注会影响正确交付的关键问题，再讨论优化项
- 若代码虽能工作，但明显存在可删除的重复、冗余防御式代码或过度工程化结构，应明确指出并给出简化方向
- 对高严重度问题要说明为什么严重，以及建议怎么处理
- 轻量主入口只保留通用判断骨架，不承载栈专项细节

## 输出物

默认输出一份轻量评审结果，建议包含：
- 结论概览：是否建议继续推进，以及一句话依据
- 高严重度：必须先处理的问题；没有则明确写无
- 中严重度：建议本轮优先修复的问题
- 低严重度 / 建议：不阻断但值得记录的优化项
- 验证与证据：本次评审参考了哪些 diff、测试、覆盖率或上下文
- 后续动作：下一步先做什么、是否需要补验证或专项复查
- 严重度口径：本次高/中/低分别基于什么标准判定，避免误把一般优化项写成阻塞项

## 结果要求

一份合格的评审结果应满足：
- 能看出结论建立在当前变更证据之上
- 能区分阻断问题、重要问题和一般建议
- 能对复杂度、重复代码、过度抽象、重复判空等常见设计问题给出清晰判断
- 能说明高/中/低的判定依据，避免严重度漂移
- 能按“结论优先、阻断优先、证据支撑、动作明确”的结构输出，便于研发快速消费
- 能指导下一步修复、复验或收口
- 保持轻量，不内嵌数据库、消息、缓存或特定语言栈的大段细则

## 可选增强

- 如涉及数据库结构、SQL、事务或数据一致性专项，可调用 `review-db-change`
- 如涉及缓存、消息、任务调度、外部依赖可靠性等专项，可调用 `review-middleware-reliability`
- 如涉及 Java 服务实现模式、分层约定或语言惯用法专项，可调用 `lang-java-service-patterns`

## 交接建议

- 若评审发现实现仍需调整，回到 `dev-build-change`
- 若评审结论已可接受，进入 `dev-finish-branch` 做最终收口
